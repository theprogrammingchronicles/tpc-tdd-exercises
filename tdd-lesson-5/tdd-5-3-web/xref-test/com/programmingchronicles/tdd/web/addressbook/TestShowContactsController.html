41   *
42   * <p>
43   * Ejemplo de Spy (Mock parcial del Objetct Under Test).<br/>
44   * No recomendado para unit testing, sintoma de "code smell", suele indicar
45   * un diseño pobre, o que no está diseñado con testing en mente.</p>

47   * <p>
48   * Pero es util usando librerías de terceros, que no controlamos,

54   * <b>RESTRICCIONES PARA EL EJERCICIO:</b><br/>
55   *   El requestDispatcher normalmente se obtiene a través del request, y no

59   *
60   * <p>

62   *     <li>- Modelo: JavaBeans (Java-Old Plain Objects).
63   *     <li>- Controlador: Un Servlet para cada petición web.
64   *     <li>- Vista: JSP (sin nada de lógica, solo JSTL + EL).
65   * </p>

117      *
118      *  <p>
119      *  Mock parcial de ShowContactsController que devuelve mocks del contenedor
120      *  de servlets, como el como el requestDispatcher.</p>
121      *
122      *  <p>
123      *  Codificación Manual de un Spy:<br/>
124      *  Se sobreescriben métodos del Object Under Test. Los métodos
125      *  getServletConfig y getServletContext deben devolver los mocks
126      *  que necesitamos para el test.</p>

132     protected class MockShowContactsController extends ShowContactsController {        
133         /**
134          * Defecto YAGNI:<br/>
135          * ¿Por que nos adelantamos proporcionando dos formas diferentes de
136          *  obtener el mock?
137          *
138          * <p>
139          * El código del controlador obtendrá el request dispatcher usando
140          * uno de los dos métodos. Haciendo un tests de caja blanca no
141          * tenemos que adelantar funcionalidad que puede que no utilicemos.</p>
142          */
143         @Override
144         public ServletConfig getServletConfig() {
145             ServletConfig servletConfig = mock(ServletConfig.class);
146             ServletContext servletContext = getServletContext();
147 
148             when(servletConfig.getServletContext()).thenReturn(servletContext);
149             return servletConfig;
150         }
151 
152         @Override
153         public ServletContext getServletContext() {
154             // El test debe programar el stub del context ya que es donde se
155             // decide la vista que se va a solicitar.
156             return mockContext;
157         }

160     /**
161      * Test del método doGet, que procesa la petición.
162      *
163      * <p>
164      * Al ser una arquitectura MVC, lo unico que hay que testear es:<br/>
165      * <ol>
166      *    <li> El controlador recibe los parametros.
167      *    <li> Construye el modelo de forma adecuada.
168      *    <li> Se redirige la petición a la vista adecuada.
169      * </ol>
170      * </p>
171      * <p>
172      * El controlador se implementará como un Servlet, y la logica se
173      * procesará en un HTTP GET.</p>
174      */
175     @Test
176     public void testDoGet() throws ServletException, IOException {
177        
178        // Se construyen mocks de los parametros del "doGet", que es el
179        // método que se quiere testear.
180        HttpServletRequest mockRequest = mock(HttpServletRequest.class);
181        HttpServletResponse mockResponse = mock(HttpServletResponse.class);
182 
183        // Programa el stub del context para que devuelva el dispatcher
184        // cuando se solicita la vista adecuada.
185        when(mockContext
186                .getRequestDispatcher(eq("/WEB-INF/views/showcontacts.jsp")))
187                .thenReturn(mockDispather);
188 
189        // Ejecución del test.
190        controller.doGet(mockRequest, mockResponse);
191 
192        // Diseño de nuestra arquitectura MVC:
193        //   El servlet (controlador) construye el modelo y lo hace
194        //   accesible a la vista, que será una JSP.
195        //
196        //   El modelo se crea como atributos del request (request.setAttribute).
197        //
198        //   En este caso aprobechamos los domain objects del sistema. Otros
199        //   controladores podrían definir modelos propios. Las necesidades
200        //   particulares de la logica de presentación no deben influir en la
201        //   lógica de negocio. El modelo es lógica de presentación, no de
202        //   negocio.
203 
204        // White Box Tests:
205        // Se verifica que el controlador llama a setAttributes con el modelo
206        // que será usado por las vistas (verificación por interacción).
207        verify(mockRequest, times(1)).setAttribute(eq("contacts"), eq(contactList));
208 
209        // Verificación por interacción de que el servlet hace un forward a la
210        // vista que muestra los contactos obtenidos.    
211        verify(mockDispather, times(1)).forward(mockRequest, mockResponse);
212 
213        // Minimizar verificaciones redundantes. Sería redundante la siguiente
214        // verificación:
215           // verify(mockContext, times(1)).getRequestDispatcher(eq("/WEB-INF/views/showcontacts.jsp"));
216        // Es una repetición de la programación del stub y por tanto se está
217        // probando indirectamente ya que el forward fallaría.
218     }
