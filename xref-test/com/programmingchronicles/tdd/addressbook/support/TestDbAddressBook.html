41  /**
42   * Test de DbAddressBook.
43   *
44   * <p><b>VALIDACION POR ESTADO vs INTERACCION</b><br/>
45   * Para independizar los test de la base de datos se deben utilizar stubs o mocks,
46   * la duda es si se puede validar usando stubbing (respuestas programadas), o
47   * mocking (interacción).</p>
48   *
49   * <p>
50   * Se aplicarán los criterios vistos en la teoría, intentando evitar lo máximo
51   * posible validaciones por interacción, y en los casos que no sea posible, la
52   * interacción verificada será la mínima requerida para satisfacer el test.</p>
53   *
54   * <p>
55   * No se debe atar la implementación al test, validando cada interacción o cada
56   * ausencia de interacción.<br/>
57   * En una validación por interacción el "verify" debe ser el equivalente al
58   * "assert". Un "assert" comprueba los resultados obtenidos, y un "verify"
59   * puede ser su equivalente, en este caso el resultado es una interacción.<p>
60   *
61   * <p><b>
62   * La excesiva verificación de interacción o verificación de ausencia de
63   * interacción hacen muchas validaciones redundantes, los tests inflexibles y
64   * poco mantenibles.</b></p>
65   *
66   * <p>
67   * <b> Ver: </b>
68   * <a href="http://monkeyisland.pl/2008/04/26/asking-and-telling/"></a>
69   * <a href="http://codebetter.com/blogs/aaron.jensen/archive/2008/04/03/separate-stub-and-verify-duplicate-code-necessarily.aspx"></a>
70   * </p>
71   *
72   * @author Pedro Ballesteros <pedro@theprogrammingchronicles.com>
73   */

101     /**
102      * Valida que se añade correctamente un contacto simple.
103      *
104      * <p>
105      * Parece que la única forma de validar que se añade bien un contacto es
106      * realizar un "verify" de {@link AddressBookDao#addContact(Contact) }.</p>
107      *
108      * <p>
109      * En este caso estamos ejercitando un método del DAO que devuelve datos
110      * que el Object Under Test utiliza, por tanto es más eficiente crear un
111      * stub de {@link AddressBookDao#addContact(Contact) } que devuelva datos
112      * útiles, la interacción se está por tanto comprobando de forma indirecta
113      * y el verify sería redundante.</p>
114      *
115      * <p>
116      * El resultado final de este método es guardar el contacto en la base de
117      * datos, y la forma de comprobar esto es asegurar que llama al DAO
118      * encargado de realizar esta tarea. Pero ¿que aportaría validar tan solo
119      * que se ha realizado la llamada al DAO?</p>
120      *
121      * <p>
122      * Mediante stubs probamos ambos casos usando validación por estado.</p>
123      *
124      * <p>
125      * La única forma de evitar la validación por interacción de forma completa
126      * sería disponer de un Fake de DAO, que permitiera añadir y recuperar
127      * datos, o usar la dependencia real.</p>
128      */
129     @Test
130     public void testAddSimpleContact() {
131         // Test Object
132         expectedContact.setFirstName("Pedro");
133 
134         // Se crea un stub para addContact del DAO, el método a probar necesita
135         // una respuesta adecuada.
136 
137         // ¿Por qué usamos any() como matcher?
138         // Se profundizará sobre esto en los siguientes tests.
139         when(mockAddressBookDao.addContact(any(Contact.class))).thenReturn("expectedId");
140 
141         // Test
142         String contactId = addressBook.addContact(expectedContact);
143 
144         // Verificación de que se obtiene el id proporcionado por el DAO. Esta
145         // verificación prueba indirectamente que se está obteniendo mediante
146         // una llamada a "addContact", ya que se obtiene el id programado.
147         assertEquals("expectedId", contactId);
148 
149         // Ahora un verify sería redundante, sería repetir exactamente lo mismo
150         // que en el when. La interacción está probada ya que se obtiene la
151         // la respuesta programada en el stub.
152 
153         // verify(mockAddressBookDao).addContact(eq(expectedContact));
154 
155         // El verify es prácticamente una copia del when.
156     }

159      * Probar que se devuelve correctamente un contacto de la base de datos.
160      *
161      * <p>
162      * Al no disponer de base de datos se simula su contenido programando
163      * respuesas del DAO (stub). Las respuesas del stub del DAO son nuestra
164      * base de datos, programar respuestas es como simular la base de datos.</p>
165      */

170         when(mockAddressBookDao.getContact("idcontact")).thenReturn(expectedContact);

172         // Test

174

176

178         // "expectedContact", no podemos probar que se devuelve exactamente
179         // esta instancia, la lógica de negocio podría hacer por ejemplo una
180         // copia para devolverla, y es mejor no fijar en el test que addContact
181         // devuelve exactamente la misma instancia.
182         assertEquals("Pedro", actualContact.getFirstName());

198      * Prueba que se puede añadir un contacto completo.
199      *
200      * <p>
201      * Ahora el test {@link #testAddSimpleContact} es redundante ya que prueba
202      * prácticamente lo mismo que este test. Es mejor mantener este testen lugar
203      * de {@link #testAddSimpleContact} ya que aquí probamos además que no
204      * hay problemas cuando el contacto es correcto (no lanza excepciones).</p>
205      *
206      *
207      * @throws ParseException
208      */
209     @Test
210     public void testAddFullContact() throws ParseException {
211 
212         // Contacto Completo de Ejemplo.
213         expectedContact.setFirstName("Pedro");
214         expectedContact.setSurname("Ballesteros");
215         Date expectedBirthday = dateFormat.parse("8/1/1974");
216         expectedContact.setBirthday(expectedBirthday);
217         expectedContact.setPhone("610101010");
218 
219         // Programación del stub para verificar que se obtiene un ID.
220         // NOTA:
221         //   Si solo hicieramos verificación por interacción, no haría falta
222         //   programar el stub.
223         when(mockAddressBookDao.addContact(any(Contact.class))).thenReturn("fullContactId");
224 
225         // Test
226         String id = addressBook.addContact(expectedContact);
227 
228         // Verificación de que se devuelve el id correcto.
229         assertEquals("fullContactId", id);
230 
231         // Ahora sí conviene asegurarse que el contacto se añade bien a la base
232         // de datos. Y esto se realiza mediante validación por interacción.
233 
234         // NO se va a verificar que se llama al DAO con la instancia "expectedContact"
235         // ya que la lógica de negocio no tendría porque entregar al DAO exactamente
236         // el mismo objeto, podría hacer una copia.
237 
238         // Como la implementación la realizamos con TDD nosotros implementamos
239         // lo que establezcamos con el test, nosotros podemos decidir que
240         // la implementación use exactamente la misma instancia (lo que es muy
241         // habitual). Pero esto hace el test más inflexible. Y así se aprovecha
242         // el ejercicio para ver ejemplos más avanzados.
243 
244         // Verificación capturando los argumentos.
245         ArgumentCaptor<Contact> argument = ArgumentCaptor.forClass(Contact.class);
246         verify(mockAddressBookDao).addContact(argument.capture());
247         assertEquals("Pedro", argument.getValue().getFirstName());
248         assertEquals("Ballesteros", argument.getValue().getSurname());
249         assertEquals(expectedBirthday, argument.getValue().getBirthday());
250         assertEquals("610101010", argument.getValue().getPhone());
251 
252 
253         // Verificación usando el eq() matcher, que compara los objetos
254         // utilizando el método "equals".
255 
256         // PELIGRO:
257         //  Por defecto el equals siempre hace una comparación de instancias.
258         //  Si no se ha sobrescrito el Contact#equals, no estaremos haciendo
259         //  la misma verificación que en el ejemplo anterior.
260 
261         //  TO DO: Se debería implementar el equals si se quiere hacer
262         //         el matching por valor en lugar de referncia usando eq().
263         //
264         //  En un ejercicio posterior se verán ejemplos de custom matchers como
265         //  alternativa a la implementación de equals.
266 
267         verify(mockAddressBookDao).addContact(eq(expectedContact));
268 
269 
270         // La verificación del eq() matcher es la verificación por defecto si
271         // no se explicita un matcher.
272 
273         // Esto es por tanto exactamente lo mismo que lo anterior.
274         verify(mockAddressBookDao).addContact(expectedContact);
275     }

584     /**
585      * Este test sería prácticamente innecesario si tenemos una batería
586      * de test de integración.
587      *
588      * <p>
589      * Es una delegación directa de la dependencia. Aun así se implementa para
590      * que sirva de ejercicio de ejemplo de comprobación de listas, e introducción
591      * a los hamcrest matchers.</p>
592      * 
593      * @throws ParseException
594      */
595     @Test
596     public void testGetAll() throws ParseException {
597         Date expedtedBirthday = dateFormat.parse("8/1/1974");
598 
599         Contact c1 = new Contact();
600         c1.setFirstName("Pedro");
601         c1.setSurname("Ballesteros");
602         c1.setBirthday(expedtedBirthday);
603         c1.setPhone("610101010");
604 
605         Contact c2 = new Contact();
606         c2.setFirstName("Eduardo");
607         c2.setSurname("Ballesteros");
608         c2.setBirthday(expedtedBirthday);
609         c2.setPhone("610101010");
610 
611         // Simulamos que la agenda tiene ambos contactos.
612         when(mockAddressBookDao.getAll()).thenReturn(Arrays.asList(c1, c2));
613 
614         // Test
615         List<Contact> contacts = addressBook.getAll();
616 
617         // Comprueba que se devuelve una lista conteniendo únicamente
618         // los dos elementos de prueba en cualquier orden.
619         assertEquals(2, contacts.size());
620         assertTrue(contacts.containsAll(Arrays.asList(c2, c1)));
621 
622         // La misma comprobación que "containsAll" pero usando los "hamcrest
