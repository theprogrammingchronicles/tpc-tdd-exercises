41  /**
42   * Test de DbAddressBook.
43   *
44   * <p>Como la implementación de DbAddressBook es una copia exacta de MemAddressBook,
45   * usaremos TDD para comenzar su refactorización.</p>
46   *
47   * <p>La mayor parte de los test de TestMemAddressBook siguen siendo válidos,
48   * por tanto se irán copiando incrementalmente, refactorizando antes de pasar
49   * al siguiente test. Identificando durante la refactorización si la nueva
50   * implementación requiere test particulares.</p>
51   *
52   * <p><b>VALIDACION POR ESTADO vs INTERACCION</b><br/>
53   * Para independizar los test de la base de datos se deben utilizar stubs o mocks,
54   * la duda es si se puede validar usando stubbing (respuestas programadas), o
55   * mocking (interacción).</p>
56   *
57   * <p>
58   * Se aplicarán los criterios vistos en la teoría, intentando evitar lo máximo
59   * posible validaciones por interacción, y en los casos que no sea posible, la
60   * interacción verificada será la mínima requerida para satisfacer el test.</p>
61   *
62   * <p>
63   * No se debe atar la implementación al test, validando cada interacción o cada
64   * ausencia de interacción.<br/>
65   * En una validación por interacción el "verify" debe ser el equivalente al
66   * "assert". Un "assert" comprueba los resultados obtenidos, y un "verify"
67   * puede ser su equivalente, en este caso el resultado es una interacción.<p>
68   *
69   * <p><b>
70   * La excesiva verificación de interacción o verificación de ausencia de
71   * interacción hacen muchas validaciones redundantes, los tests inflexibles y
72   * poco mantenibles.</b></p>
73   *
74   * <p>
75   * <b> Ver: </b>
76   * <a href="http://monkeyisland.pl/2008/04/26/asking-and-telling/"></a>
77   * <a href="http://codebetter.com/blogs/aaron.jensen/archive/2008/04/03/separate-stub-and-verify-duplicate-code-necessarily.aspx"></a>
78   * </p>
79   *
80   * @author Pedro Ballesteros <pedro@theprogrammingchronicles.com>
81   */

109     /**
110      * Valida que se añade correctamente un contacto simple.
111      *
112      * <p>
113      * Parece que la única forma de validar que se añade bien un contacto es
114      * realizar un "verify" de {@link AddressBookDao#addContact(Contact) }.</p>
115      *
116      * <p>
117      * En este caso estamos ejercitando un método del DAO que devuelve datos
118      * que el Object Under Test utiliza, por tanto es más eficiente crear un
119      * stub de {@link AddressBookDao#addContact(Contact) } que devuelva datos
120      * útiles, la interacción se está por tanto comprobando de forma indirecta
121      * y el verify sería redundante.</p>
122      *
123      * <p>
124      * El resultado final de este método es guardar el contacto en la base de
125      * datos, y la forma de comprobar esto es asegurar que llama al DAO
126      * encargado de realizar esta tarea. Pero ¿que aportaría validar tan solo
127      * que se ha realizado la llamada al DAO?</p>
128      *
129      * <p>
130      * Mediante stubs probamos ambos casos usando validación por estado.</p>
131      *
132      * <p>
133      * La única forma de evitar la validación por interacción de forma completa
134      * sería disponer de un Fake de DAO, que permitiera añadir y recuperar
135      * datos, o usar la dependencia real.</p>
136      */
137     @Test
138     public void testAddSimpleContact() {
139         // Test Object
140         expectedContact.setFirstName("Pedro");
141 
142         // Se crea un stub para addContact del DAO, el método a probar necesita
143         // una respuesta adecuada.
144 
145         // ¿Por qué usamos any() como matcher?
146         // Se profundizará sobre esto en los siguientes tests.
147         when(mockAddressBookDao.addContact(any(Contact.class))).thenReturn("expectedId");
148 
149         // Test
150         String contactId = addressBook.addContact(expectedContact);
151 
152         // Verificación de que se obtiene el id proporcionado por el DAO. Esta
153         // verificación prueba indirectamente que se está obteniendo mediante
154         // una llamada a "addContact", ya que se obtiene el id programado.
155         assertEquals("expectedId", contactId);
156 
157         // Ahora un verify sería redundante, sería repetir exactamente lo mismo
158         // que en el when. La interacción está probada ya que se obtiene la
159         // la respuesta programada en el stub.
160 
161         // verify(mockAddressBookDao).addContact(eq(expectedContact));
162 
163         // El verify es prácticamente una copia del when.
164     }

206      * Prueba que se puede añadir un contacto completo.
207      *
208      * <p>
209      * Ahora el test {@link #testAddSimpleContact} es redundante ya que prueba
210      * prácticamente lo mismo que este test. Es mejor mantener este testen lugar
211      * de {@link #testAddSimpleContact} ya que aquí probamos además que no
212      * hay problemas cuando el contacto es correcto (no lanza excepciones).</p>
213      *
214      *
215      * @throws ParseException

217     @Test
218     public void testAddFullContact() throws ParseException {
219 
220         // Contacto Completo de Ejemplo.
221         expectedContact.setFirstName("Pedro");
222         expectedContact.setSurname("Ballesteros");
223         Date expectedBirthday = dateFormat.parse("8/1/1974");
224         expectedContact.setBirthday(expectedBirthday);
225         expectedContact.setPhone("610101010");
226 
227         // Programación del stub para verificar que se obtiene un ID.
228         // NOTA:
229         //   Si solo hicieramos verificación por interacción, no haría falta
230         //   programar el stub.
231         when(mockAddressBookDao.addContact(any(Contact.class))).thenReturn("fullContactId");
232 
233         // Test
234         String id = addressBook.addContact(expectedContact);
235 
236         // Verificación de que se devuelve el id correcto.
237         assertEquals("fullContactId", id);
238 
239         // Ahora sí conviene asegurarse que el contacto se añade bien a la base
240         // de datos. Y esto se realiza mediante validación por interacción.
241 
242         // NO se va a verificar que se llama al DAO con la instancia "expectedContact"
243         // ya que la lógica de negocio no tendría porque entregar al DAO exactamente
244         // el mismo objeto, podría hacer una copia.
245 
246         // Como la implementación la realizamos con TDD nosotros implementamos
247         // lo que establezcamos con el test, nosotros podemos decidir que
248         // la implementación use exactamente la misma instancia (lo que es muy
249         // habitual). Pero esto hace el test más inflexible. Y así se aprovecha
250         // el ejercicio para ver ejemplos más avanzados.
251 
252         // Verificación capturando los argumentos.
253         ArgumentCaptor<Contact> argument = ArgumentCaptor.forClass(Contact.class);
254         verify(mockAddressBookDao).addContact(argument.capture());
255         assertEquals("Pedro", argument.getValue().getFirstName());
256         assertEquals("Ballesteros", argument.getValue().getSurname());
257         assertEquals(expectedBirthday, argument.getValue().getBirthday());
258         assertEquals("610101010", argument.getValue().getPhone());
259 
260 
261         // Verificación usando el eq() matcher, que compara los objetos
262         // utilizando el método "equals".
263 
264         // PELIGRO:
265         //  Por defecto el equals siempre hace una comparación de instancias.
266         //  Si no se ha sobrescrito el Contact#equals, no estaremos haciendo
267         //  la misma verificación que en el ejemplo anterior.
268 
269         //  TO DO: Se debería implementar el equals si se quiere hacer
270         //         el matching por valor en lugar de referncia usando eq().
271         //
272         //  En un ejercicio posterior se verán ejemplos de custom matchers como
273         //  alternativa a la implementación de equals.
274 
275         verify(mockAddressBookDao).addContact(eq(expectedContact));
276 
277 
278         // La verificación del eq() matcher es la verificación por defecto si
279         // no se explicita un matcher.
280 
281         // Esto es por tanto exactamente lo mismo que lo anterior.
282         verify(mockAddressBookDao).addContact(expectedContact);
283     }

596      * <p>

603     @Test
604     public void testGetAll() throws ParseException {
605         Date expedtedBirthday = dateFormat.parse("8/1/1974");
606 
607         Contact c1 = new Contact();
608         c1.setFirstName("Pedro");
609         c1.setSurname("Ballesteros");
610         c1.setBirthday(expedtedBirthday);
611         c1.setPhone("610101010");
612 
613         Contact c2 = new Contact();
614         c2.setFirstName("Eduardo");
615         c2.setSurname("Ballesteros");
616         c2.setBirthday(expedtedBirthday);
617         c2.setPhone("610101010");
618 
619         // Simulamos que la agenda tiene ambos contactos.
620         when(mockAddressBookDao.getAll()).thenReturn(Arrays.asList(c1, c2));
621 
622         // Test
