35   * Test de DbAddressBook.
36   *
37   * <p>Como la implementación de DbAddressBook es una copia exacta de MemAddressBook,
38   * se utilizará TDD para comenzar su refactorización</p>
39   *
40   * <p>La mayor parte de los test de TestMemAddressBook siguen siendo válidos,
41   * por tanto se irán copiando incrementalmente, refactorizando antes de pasar
42   * al siguiente test. Identificando durante la refactorización si la nueva
43   * implementación requiere test particulares.</p>
44   *
45   * <p><b>VALIDACION POR ESTADO vs INTERACCION</b><br/>
46   * Para independizar los test de la base de datos se deben utilizar stubs o mocks,
47   * la duda es si se puede validar usando stubbing (respuestas programadas), o
48   * mocking (interacción).</p>
49   *
50   * <p>
51   * Se aplicarán los criterios vistos en la teoría, intentando evitar lo máximo
52   * posible validaciones por interacción, y en los casos que no sea posible, la
53   * interacción verificada será la mínima requerida para satisfacer el test.</p>
54   *
55   * <p>
56   * No se debe atar la implementación al test, validando cada interacción o cada
57   * ausencia de interacción.<br/>
58   *
59   * En una validación por interacción el "verify" debe ser el equivalente al
60   * "assert". Un "assert" comprueba los resultados obtenidos, y un "verify"
61   * puede ser su equivalente, en este caso el resultado es una interacción.<p>
62   *
63   * <p><b>
64   * La excesiva verificación de interacción o verificación de ausencia de
65   * interacción hacen muchas validaciones redundantes, los tests inflexibles y
66   * poco mantenibles</b></p>
67   *
68   * <p>
69   * <b> Ver: </b>
70   * <a href="http://monkeyisland.pl/2008/04/26/asking-and-telling/"></a>
71   * <a href="http://codebetter.com/blogs/aaron.jensen/archive/2008/04/03/separate-stub-and-verify-duplicate-code-necessarily.aspx"></a>
72   * </p>
73   *
74   * @author Pedro Ballesteros <pedro@theprogrammingchronicles.com>
75   */

125     /**
126      * Valida que se añade correctamente un contacto simple.
127      *
128      * <p>
129      * Parece que la única forma de validar que se añade bien un contacto es
130      * realizar un "verify" de {@link AddressBookDao#addContact(Contact) }.</p>
131      *
132      * <p>
133      * En este caso estamos ejercitando un método del DAO que devuelve datos
134      * que el Object Under Test utiliza, por tanto es más eficiente crear un
135      * stub de {@link AddressBookDao#addContact(Contact) } que devuelva datos
136      * útiles, la interacción se está por tanto comprobando de forma indirecta
137      * y el verify sería redundante.</p>
138      *
139      * <p>
140      * El resultado final de este método es guardar el contacto en la base de
141      * datos, y la forma de comprobar esto es asegurar que llama al DAO
142      * encargado de realizar esta tarea. Pero ¿que aportaría validar tan solo
143      * que se ha realizado la llamada al DAO?</p>
144      *
145      * <p>
146      * Mediante stubs probamos ambos casos usando validación por estado.</p>
147      *
148      * <p>
149      * La única forma de evitar la validación por interacción de forma completa
150      * sería disponer de un Fake de DAO, que permitiera añadir y recuperar
151      * datos, o usar la dependencia real.</p>
152      */
153     @Test
154     public void testAddSimpleContact() {
155         // Test Object
156         expectedContact.setFirstName("Pedro");
157 
158         // Se crea un stub para addContact del DAO, el método a probar necesita
159         // una respuesta adecuada.
160 
161         // ¿Por qué usamos any() como matcher?
162         // Se profundizará sobre esto en los siguientes tests.
163         when(mockAddressBookDao.addContact(any(Contact.class))).thenReturn("expectedId");
164 
165         // Test
166         String contactId = addressBook.addContact(expectedContact);
167 
168         // Verificación de que se obtiene el id proporcionado por el DAO. Esta
169         // verificación prueba indirectamente que se está obteniendo mediante
170         // una llamada a "addContact", ya que se obtiene el id programado.
171         assertEquals("expectedId", contactId);
172 
173         // Ahora un verify sería redundante, sería repetir exactamente lo mismo
174         // que en el when. La interacción está probada ya que se obtiene la
175         // la respuesta programada en el stub.
176 
177         // verify(mockAddressBookDao).addContact(eq(expectedContact));

179         // El verify es prácticamente una copia del when.
