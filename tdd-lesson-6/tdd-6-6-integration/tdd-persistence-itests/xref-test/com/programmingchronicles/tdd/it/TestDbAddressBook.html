44   * Test de DbAddressBook.
45   *
46   * IMPORTANTTE: SON TEST DE INTEGRACIOÇN el Beforeclass y affet desaparecen
47   * ya no hay que crear la base de datos.
48   *
49   * <p>Como la implementación de DbAddressBook es una copia exacta de MemAddressBook,
50   * usaremos TDD para comenzar su refactorización.</p>
51   *
52   * <p>La mayor parte de los test de TestMemAddressBook siguen siendo válidos,
53   * por tanto se irán copiando incrementalmente, refactorizando antes de pasar
54   * al siguiente test. Identificando durante la refactorización si la nueva
55   * implementación requiere test particulares.</p>
56   *
57   * <p><b>VALIDACION POR ESTADO vs INTERACCION</b><br/>
58   * Para independizar los test de la base de datos se deben utilizar stubs o mocks,
59   * la duda es si se puede validar usando stubbing (respuestas programadas), o
60   * mocking (interacción).</p>
61   *
62   * <p>
63   * Se aplicarán los criterios vistos en la teoría, intentando evitar lo máximo
64   * posible validaciones por interacción, y en los casos que no sea posible, la
65   * interacción verificada será la mínima requerida para satisfacer el test.</p>
66   *
67   * <p>
68   * No se debe atar la implementación al test, validando cada interacción o cada
69   * ausencia de interacción.<br/>
70   * En una validación por interacción el "verify" debe ser el equivalente al
71   * "assert". Un "assert" comprueba los resultados obtenidos, y un "verify"
72   * puede ser su equivalente, en este caso el resultado es una interacción.<p>
73   *
74   * <p><b>
75   * La excesiva verificación de interacción o verificación de ausencia de
76   * interacción hacen muchas validaciones redundantes, los tests inflexibles y
77   * poco mantenibles.</b></p>
78   *
79   * <p>
80   * <b> Ver: </b>
81   * <a href="http://monkeyisland.pl/2008/04/26/asking-and-telling/"></a>
82   * <a href="http://codebetter.com/blogs/aaron.jensen/archive/2008/04/03/separate-stub-and-verify-duplicate-code-necessarily.aspx"></a>
83   * </p>
84   *
85   * @author Pedro Ballesteros <pedro@theprogrammingchronicles.com>
86   */

110     @Before
111     public void setUp() throws SQLException {
112         connection = newConnection();
113 
114         // Se utiliza un datasource que devuelve siempre la misma conexión
115         // y que ignora la gestión de transacciones y cierre de conexiones,
116         // para que los tests puedan realizar el rollback de las modificaciones.
117         DataSource dataSource = newSingleNonTransactionalDataSource(connection);
118 
119         // Inicializa el servicio de AddressBook para acceso a la BD de pruebas.
120         JdbcAddressBookDao dao = new JdbcAddressBookDao();
121         dao.setDataSource(dataSource);
122 
123         addressBook = new DbAddressBook();
124         addressBook.setAddressBookDao(dao);
125 
126         // Contacto por defecto usado en las pruebas.
127         expectedContact = new Contact();
128 
129         // Al inicio de cada test se asegura la gestión manual de transacciones.
130         connection.setAutoCommit(false);
131     }

133     @After
134     public void tearDown() throws SQLException {
135         // Despues de cada test se realiza un rollback para conservar
136         // la independencia entre los tests.
137         try {
138             connection.rollback();
139         } finally {
140             // Como se crea una conexión en cada test se debe realizar el
141             // close de la conexión.
142             connection.close();
143         }
144     }

442     @Test
443     public void testDeleteContact()  {
444         Contact c1 = new Contact();
445         c1.setFirstName("Pedro");
446 
447         Contact c2 = new Contact();
448         c2.setFirstName("Eduardo");
449 
450         // Una agenda con un solo contacto no asegura que solo se elimine
451         // ese contacto en lugar de eliminar toda la agenda.
452         String idDeleted = addressBook.addContact(c1);
453         addressBook.addContact(c2);
454         // Se elimina el primer contacto añadido.
455         addressBook.deleteContact(idDeleted);
456 
457         // Verifica que ya solo queda un contacto.
458         List<Contact> contacts = addressBook.getAll();
459         assertEquals(1, contacts.size());
460         assertThat(contacts, hasItems(contactNoIdEq(c2)));
461 
462         // Test: Se intenta borrar de nuevo el contacto, por lo que
463         // debe lanzar la excepción definida en la interfaz.
464         try {
465           addressBook.getContact(idDeleted);
466           fail("InvalidIdException expected");
467         }
468         catch(InvalidIdException ex) {
469           assertTrue(true);
470         }
471     }
